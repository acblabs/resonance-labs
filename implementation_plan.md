# ResonanceLab Implementation Plan

## 1. Project Summary

ResonanceLab is an open-source, sound-only active acoustic sensing platform. It uses a device speaker to emit a controlled chirp or sweep, records the object's acoustic response with the device microphone, then uses digital signal processing and machine learning to infer physical properties of everyday objects.

The project avoids cameras and images entirely. The core system uses browser audio capture, controlled chirp playback, transfer-response analysis, FFT/STFT/mel-spectrogram features, reference comparison, classical ML baselines later, and eventually small neural audio models if the data earns that complexity. LLMs such as Gemini, ChatGPT, Claude, Grok, or local models are used only as an explanation, experiment-design, debugging, and lab-assistant layer.

Core promise:

```text
ResonanceLab learns acoustic fingerprints of everyday objects and tracks how they change.
```

The first open-source release should be narrow, reproducible, and impressive:

```text
Use a browser-based app to play a short chirp near a drinking glass, record the acoustic response, analyze it with a Python/FastAPI backend, estimate fill level, visualize the signal, and explain the physics.
```

The v1 user workflow should aim for an active acoustic probe generated by the device itself. However, chirp-based sensing is still an empirical bet because phone speakers may couple weakly into glass resonance modes. Tap-based excitation should remain a first-class comparison arm during the early feasibility spike until side-by-side data proves that airborne chirps work well enough on real phones.

## 2. Naming and Source of Truth

Use consistent names from the beginning.

```text
Product name: ResonanceLab
GitHub repository: resonance-labs
Python package import: resonancelab
GCP resource prefix: resonancelab
Public docs title: ResonanceLab
```

The project will be hosted on GitHub. GCP Cloud Build will connect to the GitHub repository, build container images, push them to Artifact Registry, and deploy them to Cloud Run.

GitHub is the collaboration surface. Cloud Build is the deploy-focused CI/CD surface. The public alpha should be usable on mobile by opening the Cloud Run HTTPS URL in a mobile browser, with Android Chrome as the primary mobile target. No native mobile app is required for v1.

## 3. Important v1 Decisions

### v1 Will Use

- SvelteKit for the frontend web app. Add installable PWA behavior only when service-worker and offline-state deliverables are explicit.
- Web Audio API for microphone access and chirp playback.
- AudioWorklet or Web Audio PCM capture as the preferred browser capture path.
- MediaRecorder only as a compatibility fallback, because WebM/Opus may damage acoustic detail.
- Python + FastAPI for the backend API.
- `requirements.txt` instead of `pyproject.toml` for Python dependencies.
- Cloud Run for hosting the web app and API.
- Cloud Build for deploy-focused CI/CD from GitHub to Cloud Run.
- Cloud Build checks plus local scripts/hooks for contributor-friendly lint/test feedback.
- Artifact Registry for container images.
- Cloud Storage for model artifacts, optional uploaded recordings, benchmark outputs, and curated sample datasets.
- Vertex AI Model Garden / Gemini through Cloud Run service identity and IAM, not API keys.
- Cloud Logging for runtime observability.
- IndexedDB and localStorage in the browser for v1 calibration profiles and preferences.
- JSON/JSONL manifests for dataset and experiment metadata.
- Git hooks and CI checks for basic repo hygiene, with docs-freshness enforcement introduced gradually.

### v1 Will Not Use

- Firestore.
- Cloud SQL.
- BigQuery.
- Pub/Sub.
- Dataflow.
- GKE.
- User accounts.
- Multi-user cloud sync.
- Native mobile apps.
- Camera or image input.
- Tapping as the only required end-user workflow.
- Real-time on-device inference.
- Production-grade measurement claims.
- Public dataset submission from arbitrary users until consent, licensing, abuse prevention, and retention policies are solved.
- PyTorch in the API runtime until the CNN phase actually exists.

Firestore can be revisited in v2 or v3 if the project needs user accounts, synced profiles, shared datasets, or multi-device history. For v1, avoiding Firestore keeps the architecture simpler, cheaper, more transparent, and easier for contributors to run locally.

## 4. Risk-First Sequencing

The highest risk is not Cloud Run, repo structure, or docs governance. The highest risk is whether device-generated audio can reliably distinguish fill levels under real browser and phone conditions. Chirp-based sensing is attractive because the input signal is known and repeatable, but it may be dominated by direct speaker-to-microphone bleed, room reflections, and weak airborne coupling into the glass. This must be tested before the product path is treated as settled.

Before investing deeply in platform work, run two early spikes.

### 4.1 Week 0 Measurement Spike

Goal:

```text
Prove whether an emitted chirp can produce separable acoustic features for glass fill levels, and compare it directly against tap-based excitation.
```

Suggested experiment:

- Include at least one real phone in the first spike, not only a laptop.
- Use both phone speaker/microphone and a controlled laptop setup if available.
- Test 3 to 5 drinking glasses.
- Test empty, 25%, 50%, 75%, and full.
- Use ground truth by mass, not by eye.
- Record 20 to 30 chirp responses per fill level.
- Record a matched tap-based comparison set for the same glass/fill/session.
- Record free-air reference chirps and empty-glass reference chirps to estimate direct path and room response.
- Capture fixed geometry metadata: distance from rim, phone orientation, volume setting, room, device, and browser.
- Save WAV files and JSON metadata.
- Extract FFT/STFT, spectral peaks, transfer-response features, ring-down features, and decay features.
- Plot whether the fill levels separate by glass, session, device, browser, and excitation mode.

Go/no-go signal:

- If the glass response cannot be seen above direct speaker-to-microphone path and room reflections, pause chirp-first platform work and revise the sensing method.
- If tap separates well but chirp does not, keep tap as a supported mode or narrow the public promise before investing in chirp UX.
- If chirp works only on some devices, document supported devices and keep the public demo scoped to calibrated tracking on supported browsers/phones.
- Let the spike decide the product story: active probe is preferred, but calibrated tap tracking is an acceptable fallback if phone chirps cannot produce reliable separation.
- If fill levels separate within a calibrated glass, proceed with calibration-first browser work.
- If zero-shot cross-glass generalization is weak, keep the product promise focused on calibrated tracking.

### 4.2 Browser Audio Go/No-Go Spike

Goal:

```text
Prove that browser playback and recording preserve enough acoustic information.
```

Test early on real devices:

- Desktop Chrome.
- Desktop Firefox.
- Desktop Safari if available.
- Android Chrome.
- iOS Safari.

Mobile browser target:

```text
Mobile browser -> Cloud Run web URL -> microphone permission -> chirp playback/recording -> Cloud Run API analysis
```

Android Chrome is the primary mobile target for v1. iOS Safari should be tested and documented, but it should not block the first alpha unless it becomes part of the stated support matrix.

Known browser audio issues:

- iOS Safari often requires a user gesture before AudioContext playback or recording will work.
- Browsers may ignore `autoGainControl: false`, `noiseSuppression: false`, and `echoCancellation: false`.
- Echo cancellation is especially risky because it is designed to suppress the device speaker signal, which is exactly the chirp probe.
- MediaRecorder often returns WebM/Opus, which is lossy and voice-oriented.
- Sample rate can vary by device and OS.
- Speaker output, mic placement, AGC, and OS-level processing can distort amplitude and decay features.
- Playback and recording latency must be estimated and aligned.

Preferred v1 capture path:

```text
AudioContext user gesture
  -> generate chirp buffer
  -> play chirp through device speaker
  -> record microphone stream as PCM through Web Audio or AudioWorklet
  -> encode WAV in browser
  -> upload WAV plus probe metadata to FastAPI
```

Fallback path:

```text
MediaRecorder
  -> upload WebM/Opus
  -> backend decodes with ffmpeg
  -> backend flags reduced confidence if codec or browser constraints are known to be risky
```

- Calibration profile creation.
- Local profile storage in IndexedDB.
- Optional private dataset capture mode for labeled probes, delayed until the supervised-training phase and gated by explicit operator opt-in.
- Optional LLM explanation panel.
- Hearing-comfort guardrails: volume cap, short probe duration, clear no-headphones warning, and conservative defaults.

### Routes

```text

  Project entry and demo launcher

  Main chirp probe and analysis workflow

  Local calibration profile wizard

/profiles
  Local profiles stored in IndexedDB

/physics

  Documentation entry point
```

### Active Probe Flow

```text
1. User selects a profile or starts quick scan.
2. User places device near the object using fixed geometry guidance.
3. App warns the user not to use headphones or earbuds for probing.
4. User presses Probe.
5. Browser unlocks AudioContext.
6. App records a short pre-roll.
7. App plays a controlled chirp at a safe volume.
8. App records post-roll decay.
9. App uploads WAV/PCM response plus probe metadata.
10. API returns structured analysis.
11. Browser displays estimate, confidence, signal views, and limitations.
```

### Chirp Configuration

Initial defaults should be conservative and adjustable:

```json
{
  "signal_type": "log_chirp",
  "start_hz": 500,
  "end_hz": 10000,
  "duration_ms": 500,
  "pre_roll_ms": 250,
  "post_roll_ms": 1000,
  "amplitude": 0.35,
  "fade_ms": 10
}
```

The exact frequency range should be validated during the early spike. Phone speakers and microphones may not reproduce very low or very high frequencies reliably, and speaker-to-microphone bleed may dominate the object response. Keep hearing comfort and user consent explicit. Do not probe through headphones or earbuds. Keep chirps short, faded in/out, and capped to conservative playback levels.

## 8. DSP Pipeline

The DSP pipeline should be designed around the known emitted chirp.

```text
recorded audio + chirp metadata
  -> validate duration, sample rate, and channels
  -> decode to float PCM
  -> mono conversion
  -> preserve native sample rate when practical, usually 44.1 kHz or 48 kHz
  -> estimate noise floor from pre-roll
  -> align recording to emitted chirp with cross-correlation or matched filtering
  -> estimate direct speaker-to-microphone path from free-air or reference recordings
  -> isolate direct response, room response, and object response windows
  -> subtract or normalize against free-air / empty-glass reference when available
  -> optional deconvolution against reference chirp
  -> optionally average repeated chirps to improve repeatable object response
  -> bandpass filter using validated device/object range
  -> FFT/STFT and transfer-response features
  -> resonance peak detection
  -> decay and damping estimates from post-chirp response
  -> mel-spectrogram and MFCC summaries
  -> calibration and reference comparison
  -> later baseline model when supervised training is reactivated
  -> structured result JSON
```

Direct-path isolation strategy should be explicit before Phase 2:

- Fixed probe geometry guidance: distance from rim, phone orientation, and room setup.
- Free-air reference chirp to capture speaker-to-mic and room response.
- Store free-air reference per device, browser, profile, and probe configuration when possible.
- Do not require free-air calibration every session unless the environment, browser/device settings, probe configuration, or confidence signals indicate it is stale.
- Empty-glass reference chirp as the main per-profile baseline.
- Differential features relative to reference recordings.
- Late ring-down windows after the chirp to search for object resonance.
- Repeated chirps and synchronous averaging when the user can tolerate a longer probe.
- Side-by-side comparison with tap recordings during the early feasibility spike.

Core features:

- Dominant resonance peaks.
- Peak frequency shifts relative to calibration anchors.
- Spectral centroid.
- Spectral rolloff.
- Spectral bandwidth.
- Transfer-response magnitude by band.
- RMS envelope.
- Decay time.
- Exponential damping proxy.
- Zero-crossing rate.
- MFCC summary statistics.
- Energy by frequency band.
- Mel-spectrogram tensor.
- Noise floor and signal-to-noise ratio.
- Chirp alignment confidence.

Avoid downsampling too aggressively in v1. The old 16 kHz or 22.05 kHz target may discard useful resonance detail. Prefer keeping native 44.1 kHz or 48 kHz for raw analysis, then derive lower-rate features only if they are validated.

## 9. Calibration-First Product Path

Calibration is central to a credible v1.

Absolute zero-shot fill estimation across unknown glasses, phones, rooms, and speaker/microphone combinations is likely fragile. A calibrated relative estimate is more honest and more useful.

Primary v1 promise:

```text
Calibrate your glass with empty, 50%, and full reference chirps. ResonanceLab tracks later changes relative to that profile.
```

Calibration workflow:

```text
record or reuse free-air reference chirp
record empty anchor with chirp
record 50% anchor with chirp
record full anchor with chirp
store anchor features locally
store free-air reference locally and mark it stale when device/browser/probe settings change
compare new chirp response to anchors
estimate relative fill level
return confidence, nearest anchor, and direct-path caveats
```

Calibration should move earlier than the original Phase 5. A simple three-anchor calibration mode can produce a believable demo before a large ML dataset exists. Free-air calibration should be treated as a reusable reference, not a mandatory every-session chore, so the DSP gets useful subtraction data without making the UX feel heavy.

## 10. Phase 4 Reference Comparison and Material Hypotheses

Phase 4 should now answer a narrower and more useful question before supervised learning:

```text
Given a current chirp probe, free-air reference, local calibration anchors, and one or more known
object references, what changed in the acoustic response and what material or object hypotheses are
physically plausible?
```

This phase should not require a large labeled dataset or a trained scikit-learn/XGBoost model. It
should use the machine-listening math already in the app, then let an LLM explain structured evidence
and suggest the next measurement.

Reference library:

- Free-air reference: same device, browser, volume, probe settings, and room position without the
  object in the measurement position.
- Empty known-object reference: same geometry with a known object such as the empty glass.
- Optional state references: 50%, full, or other known states for the same object.
- Optional known-material references: glass, ceramic, plastic, metal, wood, or cardboard examples,
  recorded with the same protocol when available.
- Every reference stores probe settings, browser capture signature, quality metrics, warnings, and a
  compact DSP feature vector. Raw audio remains local/private unless explicit capture is enabled.

Comparison method:

- Reject or downweight comparisons when sample rate, browser family, capture path, probe settings,
  forced audio-processing flags, or volume setting differ from the reference.
- Compute free-air dominance first. If the current probe is closer to free-air than to object
  references, report "no strong object response" instead of forcing a material claim.
- Compare feature vectors with physically meaningful coordinates:
  - log-Hz distances for dominant peak and resonance summaries.
  - dB deltas for transfer-response and band-energy features.
  - normalized differences for spectral centroid, rolloff, bandwidth, decay, RT60 proxy, Q-factor,
    alignment, and SNR.
  - missing-feature penalties instead of silent zero fill.
- Use reliability weights from alignment confidence, SNR, anchor/reference repeat stability, capture
  compatibility, and whether a feature family is expected to be robust for the device.
- Produce distance-to-free-air, distance-to-empty, distance-to-known-state, nearest-reference,
  margin-to-runner-up, and stability fields in structured JSON.
- Treat subtraction as experimental. Prefer log-ratio/delta features and late-window ring-down
  comparisons before any claim of deconvolution. Do not present magnitude subtraction as validated
  cancellation of speaker-to-microphone bleed.

Physics interpretation:

- Glass-like thin resonators usually show sharper resonant peaks and measurable fill-dependent peak
  shifts when coupling is strong enough.
- Plastic or damped containers may show broader peaks, faster decay, and weaker Q.
- Metal may show sharper high-Q ringing, but phone speakers/mics and room reflections can mask it.
- Soft or porous materials may look closer to free-air with broadband damping rather than isolated
  resonance.
- Similarity is not identity. The output should say "most similar to the saved glass reference under
  this setup" rather than "this is glass" unless a later supervised benchmark supports that claim.

LLM role:

- The LLM receives only structured summaries: quality metrics, reference distances, peak shifts,
  band deltas, warnings, and protocol metadata.
- It may generate a human-readable hypothesis, caveats, and a suggested next experiment.
- It must not receive raw audio, claim benchmark accuracy, or override hard DSP confidence gates.

Implemented status:

- Browser-local known-object references and weighted reference comparison are implemented in the Lab
  UI.
- `/api/v1/explain` accepts analysis JSON plus compact calibration/reference summaries and rejects
  raw-audio opt-in attempts by schema.
- The Lab UI can request an explanation after a probe and receives deterministic DSP/reference
  summaries when the hosted LLM is disabled.
- Hosted Gemini explanations are configured for `gemini-3.1-pro-preview`, `global`, and `HIGH`
  thinking level, but remain disabled by default through `RESONANCELAB_LLM_ENABLED=false`.
- The normal LLM path sends no WAV bytes, full FFT/STFT/mel grids, or full local IndexedDB profile
  contents.

Exit criteria:

- The app can compare a current probe against free-air and at least one known-object reference.
- Results identify when the measurement is dominated by free-air or room response.
- Known-state comparisons are stable across repeated probes in the same geometry.
- The LLM explanation is grounded in structured evidence and clearly separates observation,
  hypothesis, and recommended next measurement.
- Supervised dataset capture remains deferred until this reference workflow produces repeatable
  separations worth labeling at scale.

## 11. Deferred Supervised ML Plan

### First Supervised Model

Use interpretable baselines first:

```text
DSP features -> Logistic Regression / Random Forest / XGBoost
```

Predict buckets:

```text
empty
25%
50%
75%
full
```

Also compute regression:

```text
fill_percent: 0 to 100
```

### Evaluation Requirements

"Better than chance" is not enough.

Use splits that prevent leakage:

- Split by recording session.
- Split by glass when testing cross-glass generalization.
- Split by device when testing cross-device generalization.
- Split by browser when browser audio behavior is part of the dataset.
- Avoid random tap/probe-level splits that put near-duplicates in train and test.

Suggested alpha targets:

- Calibrated same-glass MAE below 15% to 20%.
- Within one bucket at least 90% of the time for calibrated profiles.
- Report cross-glass and cross-device results separately.
- Always include confidence and limitations.

### Later Neural Model

Only after the baseline and dataset justify it:

```text
mel-spectrogram or transfer-response image -> small CNN -> fill-level head
```

Do not add PyTorch to the API image until this phase is active. Keep the early Cloud Run API lighter with numpy/scipy/scikit-learn/XGBoost only. Prefer direct numpy/scipy DSP for v1 when practical; add librosa only when it provides enough value to justify import time, image size, and Cloud Run cold-start cost.

### Later Multi-Task Model

Later:

```text
features or spectrogram
  -> shared encoder
  -> material head
  -> fill-level head
```

## 12. Data Persistence Strategy

Since Firestore is removed from v1, persistence is split into three simple layers.

### 12.1 Browser Local Storage

Use IndexedDB for:

- Calibration profiles.
- Anchor features.
- Optional anchor audio blobs.
- Recent analysis sessions.
- Cached result JSON.
- Local-only experiment history.

Use localStorage only for lightweight UI preferences:

- Theme.
- Last selected mode.
- API endpoint override.
- LLM enabled/disabled setting.

### 12.2 Cloud Storage

Use Cloud Storage for file-like artifacts only.

Suggested buckets:

```text
resonancelab-model-artifacts-dev
  XGBoost models
  scikit-learn models
  ONNX exports later
  model cards
  benchmark metrics

resonancelab-audio-samples-dev
  private development recordings
  not public by default

resonancelab-experiment-artifacts-dev
  generated spectrograms
  benchmark reports
  confusion matrices
  evaluation outputs

resonancelab-public-samples
  later public curated sample dataset
  only consented and licensed examples
```

In v1, the backend should not require a database to run. If a recording is uploaded for analysis, the default behavior should be ephemeral processing. Store audio only when explicitly enabled by a developer flag or clear user opt-in.

Private dataset capture automation is deferred until the supervised-training phase. It must remain
explicitly enabled, never the default behavior of ordinary probing. When enabled for a private lab
deployment:

- Cloud Run capture uses the existing `resonancelab-web` and `resonancelab-api` services. The web
  service sets `PUBLIC_PHASE4_CAPTURE_ENABLED=true`; the API sets
  `PHASE4_CAPTURE_ENABLED=true` and points at the private GCS inbox.
- The operator bearer token should be loaded from Secret Manager into the capture API runtime, not
  stored as a plaintext Cloud Build substitution, source file, or committed environment value.
- The API runtime identity should have private bucket write IAM and operator-token secret access
  only during a deliberate operator campaign. The ordinary public demo deployment should not keep
  private capture bucket write access.
- The browser collects the ground-truth fill label, session ID, glass ID, device ID, browser ID,
  room ID, volume setting, repeat index, and operator notes before or immediately after a probe.
- Capture-time validation must require the manifest's label and context minimums before upload:
  `label.fill_percent` or mass fields sufficient to derive it, plus `session_id`, `glass_id`,
  `device_id`, `browser_id`, and `room_id`. Browser/device fields may be prefilled, but the operator
  must confirm the ground truth label.
- `Save repeat` may offer a second private action to upload a manifest-ready dataset record, but the
  normal Phase 3 calibration save still remains browser-local by default.
- `Export` remains a local calibration backup unless the user explicitly chooses a private dataset
  upload/export flow.
- The browser sends the analysis JSON and an operator preference for raw WAV storage to the API; the
  server-side `PHASE4_CAPTURE_STORE_RAW_AUDIO` policy is authoritative. The browser never writes
  directly to Cloud Storage and never receives service account credentials.
- The API writes private capture artifacts only to a mutable inbox prefix, such as
  `phase4/inbox/<session_id>/audio/`, `phase4/inbox/<session_id>/analysis/`, and
  `phase4/inbox/<session_id>/records/`.
- The API stages one manifest-compatible record fragment per capture, validates the fragment before
  publishing it, and writes the fragment last so partial artifact failures do not create committed
  records. Fragment paths must be POSIX paths relative to the future finalized `manifest.json`, such as
  `audio/<session_id>/<record_id>.wav` and
  `analysis/<session_id>/<record_id>.analysis.json`; never store `gs://` URLs in manifest records.
- Capture records should persist continuous fill labels, mass fields, and raw quality metrics. The
  finalized manifest derives `fill_bucket` from its own bucket schema, and training/benchmark runs
  apply SNR/alignment inclusion thresholds instead of freezing those decisions at capture time.
- A manual "Finalize Dataset + Train" operator step materializes an immutable snapshot, such as
  `phase4/datasets/2026-06-19/` or `phase4/datasets/<build-id>/`, copies the selected inbox
  artifacts into snapshot-local `audio/`, `analysis/`, and `features/` paths, validates
  `manifest.json`, and invokes the private Cloud Build Phase 4 pipeline with
  `_PRIVATE_INPUT_PREFIX` pointed at that exact snapshot.
- `phase4/datasets/current` may exist only as a pointer or copy of the latest finalized snapshot. It
  must not receive live captures and must not be mutated while a training build is reading it.
- Start with a manual finalization/training action. Eventarc on finalized manifest objects is a
  later refinement after snapshots are immutable; do not trigger training from a mutable
  `current/manifest.json`.
- The public demo must keep dataset capture disabled unless auth, consent, abuse prevention, and
  retention policy are deliberately added.
- The private capture endpoint must have defense in depth: disabled by default, real operator auth
  when enabled, and a Cloud Run runtime identity whose private bucket write access is scoped to an
  active collection campaign.

## 13. GCP v1 Architecture

### GitHub to Cloud Run Deployment

```text
GitHub repository
  -> Cloud Build trigger
  -> install/test/lint
  -> build web and API containers
  -> push to Artifact Registry
  -> deploy to Cloud Run
```

Use Cloud Build triggers:

- Pull request trigger for tests, lint, docs checks, and container build validation.
- Main branch trigger for build and deploy to dev Cloud Run services.
- Optional tagged release trigger for production deployment later.

Current implementation status:

- `cloudbuild.yaml` supports the check/build path and an opt-in Cloud Run deployment path gated by
  `_DEPLOY_TARGET=cloud-run`.
- API and web images can be pushed to Artifact Registry and deployed to Cloud Run from Cloud Build.
- The deploy path configures runtime API URL discovery, API CORS for generated Cloud Run web URLs,
  memory, CPU, concurrency, timeout, max instances, and service names through substitutions.
- Both Cloud Run services are deployed with the second-generation execution environment and startup
  CPU boost enabled.
- Local/private GCP details are kept out of committed config through Cloud Build-provided project
  variables, ignored local gcloud config, and public-safe docs.
- Manual Cloud Build submissions have verified the Cloud Run deploy path. A GitHub-connected trigger
  still needs to be configured and validated before the main-branch deploy criterion is fully closed.

### Cloud Run Services

```text
resonancelab-web
  SvelteKit web app
  Public HTTPS frontend
  Hosts chirp probe UI, calibration UI, visualizations, and docs links

resonancelab-api
  FastAPI service
  Receives audio
  Runs DSP + inference
  Returns structured JSON
  Calls optional LLM explanation service
```

Two services are fine for v1, but plan for:

- CORS configuration.
- Environment-based API URL in the web app.
- HTTPS-only microphone access.
- Mobile access by opening the Cloud Run web URL in Chrome on Android as the primary v1 mobile browser.
- Explicit PWA installability only after service worker, manifest, update behavior, and offline/error-state expectations are defined.
- A future single-domain reverse proxy, load balancer, or SvelteKit proxy route if the web app experience needs it. For public alpha, consider same-origin API routing to reduce CORS friction.

Current Cloud Run alpha status:

- `resonancelab-web` and `resonancelab-api` exist as separate Cloud Run services.
- The web service serves the SvelteKit probe UI over public HTTPS.
- The API service accepts browser WAV uploads and returns Phase 2 DSP analysis JSON.
- The web service points at the deployed API through runtime `PUBLIC_API_URL`.
- API CORS is configured for both generated Cloud Run web URL forms.
- Desktop Chrome has been used for end-to-end manual probing against the Cloud Run deployment.
- Android Chrome and iOS Safari remain manual validation targets before claiming a mobile alpha.

### Cloud Run API Resource Defaults

The API will load DSP and ML dependencies, so start conservatively:

```text
Memory: 1 GiB initially, 2 GiB if librosa/XGBoost/model loading needs it
CPU: 1 vCPU initially
Concurrency: 1 to 4 for DSP-heavy endpoints
Max instances: capped to control cost and abuse
Min instances: 0 for low cost, optional 1 for public demo responsiveness
Request timeout: short enough to prevent abuse, long enough for DSP
```

Model loading strategy:

- Do not load models from GCS on every request.
- For alpha, either bake the baseline model into the API image or load once at startup from Cloud Storage.
- Cache loaded models in process memory.
- Keep model cards and benchmark outputs in Cloud Storage.

### Cloud Run Jobs

Cloud Run Jobs are useful later, but they should not block the early local demo.

Add jobs when the pipeline stabilizes:

```text
resonancelab-extract-features-job
  Reads audio from Cloud Storage or local mounted test data
  Extracts DSP features
  Writes JSONL/Parquet/NPY outputs

resonancelab-train-baseline-job
  Trains XGBoost or scikit-learn baseline model
  Writes model artifact to Cloud Storage

resonancelab-evaluate-job
  Runs benchmarks
  Generates metrics, plots, model cards, and reports
```

Before that, use notebooks and local scripts.

## 18. LLM Layer

The LLM should receive only structured analysis JSON, not raw audio.

Use it for:

- Result explanation.
- Reference-comparison and material-hypothesis explanation.
- Experiment design.
- Physics tutoring.
- Debugging low confidence.
- Dataset labeling help.
- Markdown report generation.

Do not use it for:

- Raw audio inference.
- Replacing the DSP/ML pipeline.
- Unsupported claims about accuracy.

Example LLM input:

```json
{
  "material_prediction": "glass",
  "fill_prediction_percent": 63,
  "fill_confidence": 0.78,
  "dominant_resonance_hz": 742,
  "decay_time_seconds": 0.38,
  "noise_floor_db": -42,
  "chirp_alignment_confidence": 0.91,
  "nearest_calibration_anchor": "50_percent",
  "reference_match": "known_glass_empty",
  "distance_to_free_air": 0.42,
  "distance_to_nearest_known_object": 0.18,
  "hypothesis_caveat": "same setup similarity, not benchmarked material classification"
}
```

LLM deployment rules:

- Off by default in public alpha.
- No LLM API keys stored for the primary GCP path.
- Local development defaults to a mock explainer so contributors can run the app without GCP access.
- Developers with GCP access can use Application Default Credentials or a configured service account to test Vertex AI locally.
- An optional non-GCP dev API key path may be added later, but it should not be required for the core app.
- Rate limited.
- Cost capped with Cloud Run max instances and application-level limits.
- No raw audio sent to the LLM.
- Hosted calls made through Vertex AI Model Garden / Gemini using Cloud Run service identity and IAM, not stored API keys.

```text
A GitHub-hosted repo where Cloud Build can test and build the project, and a browser demo where the user presses Probe, the device emits a chirp, records the response, uploads it to FastAPI, and receives structured dummy analysis JSON. The first gate is desktop Chrome; the next gate is opening the Cloud Run HTTPS URL from Android Chrome and repeating the same flow.
```

Cloud Run deployment should follow once the browser chirp path and API container are stable enough to avoid chasing deployment issues before the measurement path is real.

## 22. Final Recommendation

The revised v1 should be intentionally simple:

```text
SvelteKit web app
+ Web Audio chirp probe
+ FastAPI
+ Python DSP/ML via requirements.txt
+ GitHub
+ Cloud Build
+ Cloud Run
+ Cloud Storage
+ local IndexedDB profiles
+ Vertex AI Model Garden via Cloud Run IAM
+ gradual documentation governance
```

The key change is product honesty: ResonanceLab should start as a calibrated acoustic sensing experiment, not as a generic zero-shot object scanner. The preferred UX is a controlled chirp emitted by the device, but tap-based excitation remains a serious comparison path until the Week 0 data proves airborne chirps can beat direct-path bleed and weak coupling on real phones. Calibration makes either mode credible.

Removing Firestore from v1 is still the right call. Cloud Build to Cloud Run is a good deployment path, but it should support the measurement work rather than outrun it. The first real proof is repeatable fill tracking from real device audio; the platform exists to make that proof usable, reproducible, and eventually open to contributors.
